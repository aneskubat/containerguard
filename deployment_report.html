CRITICAL K8S001 Container runs as root (missing runAsNonRoot)
Running a container as root in Kubernetes is a critical misconfiguration. If compromised, the attacker gains root access to the container, significantly increasing the risk of a container escape to the host system.
Fix: Add to securityContext: runAsNonRoot: true runAsUser: 1000
CRITICAL K8S002 Privileged container (privileged: true)
A privileged container has access to all Linux capabilities and can directly access host devices. This is equivalent to running a process with root privileges directly on the host.
Fix: Remove 'privileged: true' from securityContext.
HIGH K8S003 Missing resource limits
Without CPU and memory limits, a single Pod can consume all node resources and bring down other services — known as a resource exhaustion attack. The Kubernetes scheduler also cannot optimally place Pods without this information.
Fix: Add to container spec: resources: limits: cpu: '500m' memory: '512Mi' requests: cpu: '250m' memory: '256Mi'
HIGH K8S004 Latest tag used for container image
The 'latest' tag is not deterministic — each deployment may run a different image version. This makes auditing difficult, prevents reliable rollbacks, and can silently introduce vulnerabilities.
Fix: Use a specific tag, e.g. nginx:1.25.3 instead of nginx:latest
HIGH K8S005 allowPrivilegeEscalation not disabled
Without 'allowPrivilegeEscalation: false', a process inside the container can gain more privileges than its parent process through setuid/setgid mechanisms, making privilege escalation attacks easier.
Fix: Add to securityContext: allowPrivilegeEscalation: false
MEDIUM K8S006 Missing liveness and readiness probes
Without health probes, Kubernetes cannot automatically detect and restart unhealthy containers. The application may be in a broken state while Kubernetes continues routing traffic to it.
Fix: Add livenessProbe and readinessProbe to the container spec.
MEDIUM K8S007 Root filesystem is not read-only
Without readOnlyRootFilesystem: true, an attacker who compromises the container can write to the filesystem — installing tools, modifying configurations, or persisting malware across restarts.
Fix: Add to securityContext: readOnlyRootFilesystem: true
MEDIUM K8S008 hostNetwork or hostPID enabled
hostNetwork: true or hostPID: true removes network/process isolation between the container and the host. An attacker can see all network interfaces and processes running on the host system.
Fix: Remove 'hostNetwork: true' and 'hostPID: true' from the Pod spec.
CRITICAL K8S010 Secrets in plaintext environment variables
Defining secrets directly as env var values means they are visible to anyone with kubectl describe access. Kubernetes Secret objects provide better access control and can be encrypted at rest.
Fix: Use secretKeyRef instead of plain values: env: - name: DB_PASSWORD valueFrom: secretKeyRef: name: myapp-secrets key: db-password
